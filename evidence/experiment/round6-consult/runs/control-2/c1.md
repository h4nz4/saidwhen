1. CONSTRAINTS AND WHY

- Single-file browser app: PocketLedger is implemented in `index.html` with embedded CSS and JavaScript. This keeps the app deployable/openable as a plain static page with no build step shown in the project.
- Browser-local persistence only: expenses are saved under `localStorage` key `pocketledger.expenses.v1`. The UI says expenses are “stored on this browser,” so the reason is local personal expense tracking without a backend/account system.
- EUR-only money model: the form says “EUR only,” formatting uses `Intl.NumberFormat(..., { currency: "EUR" })`, and CSV exports `Amount EUR`. Amounts are normalized to integer cents to avoid floating-point money drift.
- Valid expense shape is strict: saved entries must have string `id`, positive integer `amountCents`, string `category`, valid `YYYY-MM-DD` date, string `note`, and string `createdAt`. Invalid or malformed storage is ignored so corrupt local data does not break the app.
- Amounts must be greater than zero with up to two decimals: this matches currency precision and prevents zero/negative/non-money entries.
- Category is required; note is optional: the app’s form and normalization enforce category as the organizing dimension while allowing lightweight entry when no note is needed.
- Dates must be valid local calendar keys in `YYYY-MM-DD`: month totals, sorting, filtering, and stored validation all depend on string slicing/comparison of that format.
- The dashboard is current-month oriented: month total and category totals use the active month at page load. This supports the app’s “this month” spending view.
- Expense display is newest first: rows sort by descending date, then descending creation timestamp. The reason is stated in UI copy: “Newest entries first.”
- Category filtering is exact-match against saved categories: categories are derived from existing expenses and sorted alphabetically, so users filter by real ledger categories rather than a separate taxonomy.
- CSV export includes every saved expense, sorted newest first, with RFC-style escaping for commas, quotes, and newlines. This provides portable backup/export without server involvement.
- Export is disabled when there are no expenses: prevents generating an empty file and matches the tested button behavior.
- Delete is immediate by expense id: each row carries `data-delete-id`, and deletion rewrites the saved array. This keeps the local ledger simple and id-based.
- HTML escaping is required for rendered expense fields and dynamic empty-state text. The reason is user-entered category/note content is injected into `innerHTML`.
- Responsive, accessible static UI: the page has semantic labels, `aria-live` regions, screen-reader-only captions/action text, and mobile breakpoints. This keeps the single-page ledger usable on small screens and assistive tech.
