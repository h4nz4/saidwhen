The wiki is silent on CSV export itself, but two recorded constraints apply. Here's the proposal.

## 1. RECOMMENDATION

Build the smallest thing: a single server-side endpoint that streams the authenticated user's own tasks as CSV, generated with the standard library's CSV writer. No new dependency, no config, no async job queue, no cloud bucket.

- **Scope:** current user's tasks only (auth already gates this via magic links). One button → one synchronous download.
- **Implementation:** stdlib CSV writer serializing existing task fields to an HTTP response with `Content-Type: text/csv` and a `Content-Disposition` attachment header. Roughly a dozen lines.
- **Skip for now:** background/async export, S3 or emailed-file delivery, custom column pickers, Excel/`.xlsx`, PDF. Add async only if a user's task count grows large enough to time out a request — not before.

Rationale: the [solo-maintainer constraint](knowledge/constraints/solo-maintainer.md) says every choice must minimize operational surface and "anything that can break at 3am is out." A synchronous stdlib endpoint has near-zero operational surface — no queue, no storage, no credentials to rotate. That's the design the recorded constraints point to.

One caution worth flagging in the code: CSV opened in Excel is subject to formula injection (a task title starting with `=`, `+`, `-`, `@`). Prefix such fields with a `'` on export. This is the one place not to be lazy — it's a trust-boundary/output-safety issue, cheap to handle.

## 2. PRIOR CONTEXT

What in the wiki informed the above:

- **[knowledge/constraints/solo-maintainer.md](knowledge/constraints/solo-maintainer.md)** — one person, a few hours/week; minimize operational surface; nothing that breaks at 3am. This is the primary driver for the stdlib, synchronous, zero-infrastructure recommendation.
- **[knowledge/constraints/external-users.md](knowledge/constraints/external-users.md)** — users are individual freelancers with no corporate IdP. Reinforces that export is per-individual-user, not per-org/team; no multi-tenant or role-scoped export to design for.
- **[knowledge/decisions/magic-link-auth.md](knowledge/decisions/magic-link-auth.md)** (accepted 2026-07-08) — auth is already handled by magic links; the export endpoint just reuses the existing session, no new auth surface. The recorded owner posture ("don't want the liability") supports keeping the export scoped to the caller's own data.
- **[knowledge/interviews/2026-07-08-auth-scope.md](knowledge/interviews/2026-07-08-auth-scope.md)** — source for the two constraints above ("Basically none" maintenance budget).

The wiki records **no prior decision about exports, data portability, or file formats** — this is greenfield relative to the bundle. I'm not citing any decision that doesn't exist; the constraints above genuinely apply and I've carried them over.

## 3. QUESTIONS

These aren't answered by the wiki, so they're worth asking the PM/owner:

1. **Which fields go in the CSV?** All task fields, or a subset? Anything sensitive that should be excluded from a downloadable file?
2. **Scope confirmation:** just the requesting user's own tasks (my assumption), or is there any shared/team view I'm not seeing in the bundle?
3. **Expected volume** — roughly how many tasks does a heavy user have? If it's plausibly tens of thousands, that's the only thing that would justify async export instead of a synchronous download.
4. **Trigger surface:** a button in the existing UI, a CLI/API call, or both?
