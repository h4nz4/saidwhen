## 1. RECOMMENDATION

Ship the laziest version that works: **a synchronous, on-demand CSV download of the requesting user's own tasks, generated in-process with Python's stdlib `csv` module.** One route (`GET /tasks/export.csv`), one handler, no new dependency, no background jobs, no file storage.

```
GET /tasks/export.csv  → 200, Content-Type: text/csv,
                          Content-Disposition: attachment; filename="tasks.csv"
```
- Query the authenticated user's tasks, stream rows through `csv.writer` to the response.
- Scope strictly to `session.user_id` — export is a per-user data dump, same trust boundary as viewing tasks.
- `csv.writer` handles quoting/escaping/newlines correctly; do **not** hand-roll `",".join(...)` (that breaks on any comma or quote in a task title — a real injection/corruption path).

**Skipped, and when to add:**
- Async job + emailed download link → add when a single user's export can't finish inside one request. For a solo-run freelancer tracker, task counts are tiny; not now.
- Column selection / format options (XLSX, JSON, date-range filters) → add when a user actually asks. YAGNI.
- Formula-injection hardening (prefixing `=+-@` cells with `'`) → low risk here since users export *their own* data back to themselves, but it's ~3 lines. Add the moment exported data could be authored by anyone other than the recipient (shared workspaces, admin exports).

This is one rung above "does it need to exist" — the PM requested it — and lands on stdlib. No higher rung needed.
