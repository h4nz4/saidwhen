1. RECOMMENDATION

Do not rewrite PocketLedger in React yet.

PocketLedger is currently a compact static app in `index.html` with plain HTML/CSS/JS, localStorage persistence, CSV export, and focused Node-based tests in `pocketledger.test.js`. The current architecture is easy to inspect, has no install/build step, and already exposes pure ledger functions through `window.PocketLedger` for testing.

A React rewrite would add contributor-familiar tooling, but for this app’s present size it would mostly introduce maintenance surface: package management, bundler config, dependency updates, generated output, and a more complex test setup. I would instead propose an incremental “contributor readiness” phase first:

1. Add a minimal project README, contribution notes, and documented test/smoke commands.
2. Extract the inline JS/CSS into separate source files only if maintainability is already painful.
3. Add a lightweight package pipeline around the existing app if needed, for example `npm test`, formatting, and static checks.
4. Revisit React when the roadmap includes multiple screens, shared components, routing, richer state, charts, import flows, sync/accounts, or other complexity that plain DOM code will strain under.

If the PM’s goal is specifically contributor attraction, I would frame React as a future migration option, not the first move.
