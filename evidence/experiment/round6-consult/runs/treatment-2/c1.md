1. CONSTRAINTS AND WHY

- PocketLedger must remain a static browser-only app: one `index.html`, plain HTML/CSS/JS, no build step, framework, package dependency, backend, server, accounts, hosted infrastructure, or operated service. Data stays in `localStorage`.
  Reason: the owner described it as a personal tool and does not want to operate anything.

- PocketLedger is EUR-only. It should not add currency selection, conversion, multi-currency reporting, or multi-currency export columns.
  Reason: the owner only spends in EUR.

- CSV export must be browser-only and local. It should download all stored expenses from `localStorage`.
  Reason: the owner explicitly asked for a CSV export button, and local download preserves the static, disk-openable architecture.

- CSV export covers all expenses, not only the currently filtered list, and uses an `amount_eur` decimal column.
  Reason: the recorded export request says the button should download all expenses, and the accepted decision keeps the existing EUR-only money model.

- The current scope rejects CSV import, backup scheduling, account sync, sharing, hosted export storage, multi-file reporting exports, analytics/reporting exports, and multi-currency export/conversion.
  Reason: these would expand PocketLedger beyond the requested small personal local tool, or violate the accepted browser-only/EUR-only constraints.

- Revisit triggers are narrow: consider infrastructure only if sharing, syncing, managed backups, or multi-device use becomes necessary; consider multi-currency only if the owner regularly spends in another currency; consider import/backup/sharing/analytics exports only if the owner asks for them.
  Reason: the decisions explicitly preserve the current small local scope while naming conditions that would justify reconsideration.
