## 1. RECOMMENDATION

**Build it, but at the laziest tier: a synchronous, on-request CSV download generated from the tasks a user already sees — no async jobs, no email delivery, no new storage, no third-party service.**

Concretely: one endpoint (e.g. `GET /tasks.csv`) behind the existing session, that streams the caller's own tasks straight from the DB using the language's **standard-library CSV writer** (Python `csv`, Node has none but it's ~10 lines, Go `encoding/csv`, etc.). No new dependency, no new infrastructure, no operational surface. This is a few hours of work and nothing to run at 3am.

Explicitly **not** recommended for a first cut: background export jobs, scheduled/emailed reports, "export all formats," an admin bulk-export across users, or a queue. Those are speculative until a user actually asks, and each one adds exactly the maintenance surface this project has decided it cannot afford.

## 2. PRIOR CONTEXT (what informed this)

- **`knowledge/constraints/solo-maintainer.md`** — TaskLite is built and operated by one person, a few hours/week; *"every design choice must minimize operational surface."* This is the dominant force here: it rules out async job runners, queues, and any always-on export service in favor of a synchronous in-request generator.
- **`knowledge/constraints/external-users.md`** — users are individual freelancers with no corporate IdP. Export is per-user self-service; there is no org/admin "export everyone's tasks" role to design for.
- **`knowledge/decisions/magic-link-auth.md`** (accepted 2026-07-08) — sessions are magic-link, per individual user, no OAuth/passwords. The export endpoint should sit behind that existing session and scope strictly to the caller's own rows. I'm respecting this decision, not re-litigating it.
- **The wiki is silent on export/CSV specifically** — there is no prior decision, interview, or constraint about data export. So the feature itself is an open question; only the *how* is constrained (by solo-maintainer). I am not citing any decision that doesn't exist.
- **`README.md` / repo state** — TaskLite here is a provenance-demo fixture (only `README.md` + `specs/auth/spec.md`; no task model or app code in the tree). So the field list below is a proposal, not a reading of a real schema — see the first question.

## 3. QUESTIONS (for the PM/owner)

1. **What fields, and stable how?** There's no task schema in the repo. Confirm the columns (id, title, status, due date, created/updated, tags/notes?) and whether column order/names are a **contract** freelancers will build spreadsheets against — that decides whether we can freely change them later.
2. **Scope of an export = exactly what the user currently sees?** I'm assuming caller's own tasks only, honoring any active filter/search. Confirm; also whether completed/archived tasks are included by default.
3. **What's the actual use case?** Backup, taxes/invoicing, or moving to another tool? If it's "import into X," the target tool's expected format may matter more than plain CSV.
4. **Volume ceiling?** Freelancer task counts are small (hundreds, not millions), which is *why* synchronous generation is safe. If anyone realistically holds tens of thousands of tasks, tell me now — that's the one thing that would justify streaming/async later.
5. **Excel-friendliness needed?** UTF-8 with a BOM and CRLF so accented names/emails don't mangle in Excel — trivial to include, worth a yes/no.
