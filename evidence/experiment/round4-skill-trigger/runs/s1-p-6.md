## 1. RECOMMENDATION

**Don't build it.** This exact feature was evaluated and explicitly rejected today (2026-07-08) in an `accepted` decision. Nothing has changed since — the evidence is same-day fresh, not stale — so per the project's read-first rule I respect it rather than re-litigate.

The rejection wasn't "OAuth is bad," it was that TaskLite's users have **no corporate identity provider** to SSO against, and each OAuth provider adds console setup, credential rotation, and 3am breakage surface that a **solo maintainer** can't carry. Magic-link auth already covers the actual need (passwordless sign-in, no password liability).

The decision names one reopener: *"Revisit if a significant user segment demands social login."* Absent evidence of that, the answer stands.

## 3. QUESTIONS

These target the *only* thing that could reopen the decision — I'm not re-asking what the interview already settled:

1. **Is there evidence of the reopener?** Are users actually asking for Google sign-in, or is this a PM hypothesis? What's the size of that segment?
2. **What problem is this solving that magic links don't?** If it's login friction, that's a UX tweak to the magic-link flow, not a new IdP.
3. **Who owns the ongoing OAuth ops** (Google Cloud console, consent screen verification, secret rotation) given the solo-maintainer constraint — you, or is headcount changing?

If the answer to #1 is "no hard evidence yet," my recommendation is to leave the accepted decision in place and revisit when demand is real.
