1. CONSTRAINTS AND WHY

- PocketLedger v1 must remain a plain static single-page app: one `index.html` with plain HTML/CSS/JS, working directly from disk.  
  Why: the owner chose a no-build v1; frameworks, build steps, and package dependencies were rejected to keep the app simple and maintainable.

- Expense data must stay in browser `localStorage`; no backend, server, accounts, sync, hosted database, or operations burden.  
  Why: v1 is intended to be browser-only personal storage. Server-side operations were explicitly rejected.

- The app supports EUR only.  
  Why: the owner only spends in EUR. Multi-currency was considered and rejected for v1.

- Implementation should stay small and boring.  
  Why: one person maintains PocketLedger in spare time, so clever abstractions, extra dependencies, and operational complexity are rejected unless they solve a real owner problem.

- v1 feature scope is add expense, list expenses, delete expenses, and show the current month’s total.  
  Why: this is the feature set recorded in the owner kickoff; I found no project source expanding v1 beyond that.
